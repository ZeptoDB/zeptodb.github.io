Cluster-internal TCP RPC is the nervous system of a distributed database. In ZeptoDB, every tick ingest, WAL replication, and distributed query flows through it. Without authentication, an attacker with network access can execute arbitrary SQL, inject ticks, or manipulate WAL segments. Here's how we locked it down.

## Authentication Design

The solution uses a shared-secret HMAC with nonce-based replay protection. No OpenSSL dependency вҖ” the HMAC uses a 4-round FNV-1a hash producing a 256-bit digest.

### Configuration

```cpp
RpcSecurityConfig config;
config.enabled       = true;
config.shared_secret = "my-cluster-secret-key";
// mTLS fields prepared for future use:
config.cert_path     = "/etc/zeptodb/node.crt";
config.key_path      = "/etc/zeptodb/node.key";
config.ca_cert_path  = "/etc/zeptodb/ca.crt";
```

### Auth Payload Structure

```
Auth payload (40 bytes):
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  nonce (8 bytes) в”Ӯ  HMAC-FNV1a-256 (32 bytes)   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

- `rpc_generate_nonce()` вҖ” 8 bytes of cryptographic randomness
- `rpc_compute_hmac(secret, nonce)` вҖ” 4-round FNV-1a over secret + nonce
- `rpc_build_auth(secret)` вҖ” combines nonce + HMAC into 40-byte payload
- `rpc_validate_auth(secret, payload)` вҖ” server-side verification

## Protocol Flow

Authentication happens once per connection, before any data messages:

```
Client                              Server
  в”Ӯ                                   в”Ӯ
  в”ңв”Җв”Җ connect() в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’в”Ӯ
  в”ңв”Җв”Җ AUTH_HANDSHAKE {nonce + HMAC} в”ҖвҶ’в”Ӯ
  в”Ӯ                                   в”ңв”Җв”Җ recompute HMAC from nonce + secret
  в”Ӯ                                   в”Ӯ   в”ңв”Җв”Җ match    вҶ’ AUTH_OK
  в”ӮвҶҗв”Җв”Җ AUTH_OK в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө   в””в”Җв”Җ mismatch вҶ’ AUTH_REJECT + close
  в”Ӯ                                   в”Ӯ
  в”ңв”Җв”Җ SQL_QUERY в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’в”Ӯ  (normal traffic)
  в”ӮвҶҗв”Җв”Җ SQL_RESULT в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
  в”ңв”Җв”Җ TICK_INGEST в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’в”Ӯ
  в”ӮвҶҗв”Җв”Җ TICK_ACK в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
```

Three new message types were added to the RPC protocol:

| Type ID | Name | Direction |
|---------|------|-----------|
| 15 | `AUTH_HANDSHAKE` | Client вҶ’ Server |
| 16 | `AUTH_OK` | Server вҶ’ Client |
| 17 | `AUTH_REJECT` | Server вҶ’ Client |

## Server and Client Integration

On the server side, `TcpRpcServer` enforces authentication as the first message:

```cpp
TcpRpcServer server;
server.set_security(config);
// handle_connection() now:
//   1. Waits for AUTH_HANDSHAKE as first message
//   2. Validates HMAC вҶ’ AUTH_OK or AUTH_REJECT + close
//   3. Enters normal message loop only after AUTH_OK
```

On the client side, `TcpRpcClient` authenticates new connections automatically:

```cpp
TcpRpcClient client;
client.set_security(config);
// acquire() now:
//   1. Sends AUTH_HANDSHAKE on new connections
//   2. Waits for AUTH_OK (closes on AUTH_REJECT)
//   3. Pooled connections skip re-authentication
```

## Backward Compatibility

When `security_.enabled = false` (the default), no handshake occurs. Existing clients and servers continue to work without any protocol changes. This means rolling upgrades are safe вҖ” enable security on servers first, then clients.

---

## mTLS Roadmap

The `RpcSecurityConfig` already carries `cert_path`, `key_path`, and `ca_cert_path` fields. The plan:

1. **Current** вҖ” HMAC shared-secret authentication (deployed)
2. **Next** вҖ” OpenSSL `SSL_CTX` wrapping of TCP sockets for full mTLS
3. **Defense in depth** вҖ” HMAC authentication will be retained even after mTLS, providing two independent authentication layers

The HMAC layer is lightweight (no TLS handshake overhead) and serves as a fast-fail mechanism before the more expensive TLS negotiation.

---

## Key Takeaways

- Zero external dependencies вҖ” FNV-1a HMAC avoids OpenSSL for the auth layer
- Per-connection authentication with nonce prevents replay attacks
- Backward compatible вҖ” disabled by default, safe for rolling upgrades
- mTLS-ready config structure for future hardening
