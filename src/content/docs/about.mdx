## Mission

Make live operational data usable as memory for AI agents. ZeptoDB keeps the time-series engine fast enough for real-world systems, then adds the context, cache, and replay layer agents need to act with evidence.

---

## Why we built ZeptoDB

Robots, factories, trading desks, fleets, grids, and observability platforms all produce the same kind of hard data: ordered events arriving faster than conventional databases can ingest, align, and query. AI agents are now being attached to those systems, but most agent stacks treat memory as a separate prompt artifact or vector index.

That split loses the most important thing: time.

The useful question is not only "which memory is semantically similar?" It is also:

- What happened before this decision?
- Which raw signals changed?
- What did the agent retrieve?
- Did we call a model or reuse a cache hit?
- Which action followed, and what happened afterward?

ZeptoDB closes that gap. The time-series core stores the operational timeline. The Agent Memory layer stores agent-scoped context, embeddings, cache entries, and retrieval metadata beside that timeline. Together they make agent behavior replayable, debuggable, and grounded in live data.

---

## What ZeptoDB is

- **A microsecond time-series engine** for high-throughput ingestion, temporal joins, SQL analytics, hot/cold storage, and zero-copy Python.
- **An additive Agent Memory layer** for tenant/session scoped memories, client-supplied embeddings, context assembly, and exact/semantic prompt cache lookup.
- **An AgentOps substrate** where agent runs, retrieval events, cache events, LLM calls, tool calls, and raw operational events can share one timeline.

ZeptoDB is not trying to become a generic standalone vector database. Its strength is the combination: raw events explain what happened, memory records explain what the agent learned, and SQL ties both back to time.

## Technical philosophy

- **Time first** — every fact, memory, retrieval, and action should be explainable in sequence
- **Zero-copy everywhere** — data flows from network to column store to Python without unnecessary serialization
- **Client-controlled AI boundaries** — applications provide embeddings and call model providers; ZeptoDB stores, filters, ranks, and retrieves
- **SQL, not a DSL** — standard SQL with temporal extensions for operational teams
- **Memory beside evidence** — agent memories should not drift away from the raw events that made them useful
- **Source-available core, no phone-home** — BUSL-1.1 with an Additional Use Grant for production use, plus offline-validated Enterprise feature gates
