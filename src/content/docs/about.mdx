## Mission

Make microsecond-latency time-series the default substrate for anyone building a real-time system — from a humanoid startup in a garage to a grid operator keeping the lights on.

---

## Why we built ZeptoDB

A new generation of systems has the same bottleneck, regardless of industry. A humanoid robot, a factory cell, a driverless vehicle, a smart meter network, and a trading desk all produce time-series data faster than a conventional database can ingest it, align it, and hand it to a model.

Existing options all force a trade-off:

- **kdb+** delivers microsecond performance, but costs $100K+/year and ships with a proprietary language that most teams don't want to learn in 2026.
- **ClickHouse, TimescaleDB, InfluxDB** are affordable, mature, and well-loved — but they run at millisecond latency and were not designed for the closed-loop and tick-by-tick workloads emerging across Physical AI, autonomy, and industrial control.
- **None** offer a 522ns zero-copy path into Python, which is where the model that consumes the data actually lives.

ZeptoDB closes that gap. Microsecond performance. Standard SQL. Zero-copy to NumPy and PyTorch. Open source at the core, with a sensible Enterprise edition for clustering, SSO, and audited feature gating.

---

## Technical philosophy

- **Zero-copy everywhere** — data flows from network to column store to Python without copying
- **LLVM JIT** — queries compile to native code, not interpreted
- **SIMD-first** — Highway library for portable AVX2 / AVX-512 / NEON vectorization
- **Lock-free ingestion** — MPMC ring buffer, no mutex on the hot path
- **SQL, not a DSL** — standard SQL with temporal extensions (ASOF JOIN, xbar, EMA)
- **Open source core, no phone-home** — the engine is yours; Enterprise features are gated by offline-validated license
